# Data Retrieval & LLM Injection — Detailed Context

## Overview

This document covers how client and stock data flows from Supabase into the LLM's context during a call. It spans the database layer, the FastAPI endpoints, the orchestrator scripts, and the exact moment the data becomes part of the AI's conversational memory.

## Database Tables

### `public.clients`

Stores the master list of clients to call.

```sql
create table public.clients (
  id bigint generated by default as identity not null,
  name character varying not null,
  phone_number bigint not null,
  created_at timestamp with time zone not null default now(),
  constraint clients_pkey primary key (id),
  constraint clients_phone_number_key unique (phone_number)
);
```

### `public.clients_dummy_data`

Stores each client's stock purchase records for the day.

```sql
create table public.clients_dummy_data (
  id bigint generated by default as identity not null,
  created_at timestamp with time zone not null default now(),
  name character varying not null,
  phone_number bigint not null,
  stock_name character varying not null,
  quantity bigint not null,
  avg_rate double precision not null,
  stock_buy_value double precision not null,
  constraint clients_dummy_data_pkey primary key (id)
);
```

## Database Connection

**File:** `supabase/direct_supabase_connection.py`

- Uses `psycopg2` (via `psycopg2-binary`) for direct PostgreSQL connection.
- Credentials loaded from `.env`: `user`, `password`, `host`, `port`, `database`.
- `connect_postgres()` returns a `(connection, cursor)` tuple, or `False` on failure.
- **Important:** Every endpoint creates a new connection, uses it, and closes it in a `finally` block. There is no connection pooling.

## API Endpoints

**File:** `routers/clients.py`

### `GET /clients/`

- Queries `SELECT * FROM public.clients`
- Returns all clients as a list of dictionaries:
  ```json
  {"status": "success", "data": [{"id": 1, "name": "Aryan", "phone_number": 9876543210, ...}]}
  ```

### `GET /clients/dummy_data/{phone_number}`

- Queries `SELECT stock_name, quantity, avg_rate, stock_buy_value FROM public.clients_dummy_data WHERE phone_number = %s`
- Returns columns and rows separately (not as dictionaries):
  ```json
  {
    "status": "success",
    "columns": ["stock_name", "quantity", "avg_rate", "stock_buy_value"],
    "rows": [["Tata Steel Limited", 100, 172.1178, 17211.78], ...]
  }
  ```
- This format is deliberate — it's directly passed as context to the LLM.

## LLM Context Injection

### Step 1: Stock data is formatted

**Function:** `_build_stock_data_message()` in `app.py`

Converts the columns/rows into a human-readable string:

```
Client Name: Aryan
Columns: stock_name, quantity, avg_rate, stock_buy_value
Rows:
  ['Tata Steel Limited', 100, 172.1178, 17211.78]
  ['Irctc Ltd.', 19, 701.9495, 13337.04]
  ...
```

### Step 2: Injected into chat history

**Function:** `_bot_greeting()` in `app.py`

The chat history for this call is initialized with two messages:

```python
history.append({"role": "system", "content": SYSTEM_PROMPT})
history.append({
    "role": "user",
    "content": f"Here is the stock data for today's call:\n\n{stock_message}"
})
```

### Step 3: LLM generates greeting

A direct Groq API call is made with this two-message history. The LLM generates a greeting that summarizes the stocks. The response is appended as an `"assistant"` message.

### Step 4: Persistent memory

From this point on, every user utterance is appended to this same `call_histories[call_sid]` list. The LLM always sees:
1. System prompt (persona + rules)
2. Stock data (the full table)
3. Bot greeting
4. User question 1
5. Bot answer 1
6. User question 2
7. ... and so on

The stock data is never lost — it persists in the conversation history for the entire call.

## System Prompt

**File:** `groq_services/groq_llm.py`

The system prompt defines the LLM persona "Jeevan" and includes critical rules:
- Greet customer by name, summarize top stocks (don't read line-by-line)
- Max 3-4 sentences (voice call constraint)
- Language: English by default, Hinglish if user speaks it (numbers in English), Hindi if explicitly requested
- Never advise buying/selling
- All values are in Indian Rupees (INR)
- End with "You may end the call" when user wants to leave
